# 岁迹 LifeAtlas 软件需求规格说明书

## 1. 范围

本文档定义岁迹 Android App 的功能性需求、非功能性需求、数据需求和约束条件。第一阶段以 Android 单端本地优先为主，后续预留云同步和开源协作空间。

## 2. 功能性需求

### 2.1 记录管理

#### FR-001 新增记录

系统应允许用户创建一条人生记录。

必填字段：

- 标题
- 发生时间

可选字段：

- 正文
- 地点名称
- 经纬度
- 心情
- 重要程度

#### FR-002 查询记录

系统应支持按发生时间倒序查询所有记录。

#### FR-003 查看详情

系统应支持根据记录 ID 查看单条记录详情。

#### FR-004 编辑记录

系统应支持修改记录的标题、正文、时间、地点、心情和重要程度。

#### FR-005 删除记录

系统应支持删除记录，并同步删除该记录与照片、标签的关联数据。

### 2.2 时间轴

#### FR-006 时间轴展示

系统应按年份和月份分组展示记录。

#### FR-007 记录摘要

时间轴中的记录卡片应展示标题、时间、地点和正文摘要。

### 2.3 地图

#### FR-008 地图点位展示

系统应在地图上展示所有带经纬度的记录。

#### FR-009 地图点位交互

用户点击地图点位后，系统应展示记录标题、地点和时间摘要。

### 2.4 照片

#### FR-010 照片选择

系统应使用 Android Photo Picker 允许用户选择照片。

#### FR-011 照片关联

系统应将照片 URI 与记录 ID 建立关联。

#### FR-012 照片展示

系统应在记录详情页展示已关联照片。

### 2.5 标签

#### FR-013 标签创建

系统应支持创建文本标签。

#### FR-014 标签绑定

系统应支持一条记录绑定多个标签。

#### FR-015 标签筛选

系统应支持按标签筛选记录。

### 2.6 数据导出

#### FR-016 JSON 导出

系统应支持将记录、照片引用、标签和地点信息导出为 JSON 文件。

## 3. 非功能性需求

### 3.1 隐私

- NFR-001：系统默认本地优先，不强制登录。
- NFR-002：系统不得在未获得用户明确授权时上传记录、照片或定位数据。
- NFR-003：导出和删除能力应作为长期基础能力保留。

### 3.2 性能

- NFR-004：MVP 阶段本地 10,000 条记录内，时间轴首屏加载应保持流畅。
- NFR-005：图片不应直接以二进制形式存入 Room。
- NFR-006：地图点位超过一定数量后，应预留聚合或分页加载策略。

### 3.3 可维护性

- NFR-007：业务代码按 feature、data、core、navigation 分层。
- NFR-008：UI 状态与数据访问逻辑分离。
- NFR-009：核心数据访问应通过 Repository 暴露，不在 UI 层直接访问 DAO。

### 3.4 可扩展性

- NFR-010：本地数据模型应预留云同步字段，如 createdAt、updatedAt、deletedAt、syncState。
- NFR-011：地图供应商应通过适配层隔离，后续可支持高德地图和 Google Maps。
- NFR-012：导出格式应版本化，便于后续导入和兼容升级。

### 3.5 可靠性

- NFR-013：数据库升级必须提供 Room Migration。
- NFR-014：删除记录前应有用户确认。
- NFR-015：导出过程失败时应提示原因，不应破坏本地数据。

## 4. 约束条件

- 第一版仅支持 Android。
- 第一版不建设后端。
- 第一版不做自动后台定位。
- 第一版不做社交传播。
- 国内用户优先时，地图 SDK 默认选择高德地图。

## 5. 质量门槛

每个功能进入主分支前应满足：

- 可以编译
- 核心流程可手动验收
- DAO、Repository 等关键逻辑有单元测试或集成测试
- 不引入未说明的敏感权限
- 不破坏已有数据结构
