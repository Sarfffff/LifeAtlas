### P1：防恶意注册与登录安全

需要结合常见登录风险设计，不只做“能登录”。

建议策略：

- 邮箱验证码发送频率限制，例如同邮箱 60 秒内只能发送一次。
- 单邮箱每日验证码次数限制。
- 单 IP 每小时注册和验证码请求次数限制。
- 登录失败次数限制，连续失败后短时间锁定。
- 验证码有效期，例如 10 分钟。
- 验证码只存哈希，不明文保存。
- 密码使用强哈希算法，例如 Argon2id 或 bcrypt。
- 密码复杂度要求和弱密码拦截。
- 注册接口加入图形验证码或 Turnstile/hCaptcha 等人机验证。
- 设备指纹或匿名设备 ID 只用于风控，不用于过度追踪。
- 风险操作需要二次验证，例如修改邮箱、重置密码、删除账号。
- 服务端记录审计日志，便于排查异常注册和撞库攻击。

### P1：后端与云同步基础

如果接入登录，后端需要同时考虑数据同步。

建议拆分：

- 账号服务：注册、登录、邮箱验证、密码重置、Token 刷新。（已完成最小可用版）
- 用户资料服务：昵称、头像、偏好设置。
- 记录同步服务：记录、标签、照片元数据同步。（已完成手动轻量备份，未做实时多端合并）
- 媒体服务：照片缓存或原图对象存储。
- 同步冲突策略：本地优先、服务端版本号、最后修改时间、软删除。
- 数据导出能力保持可用，避免用户被云端锁定。

## 4. 技术选型建议

### 后端

推荐优先考虑：

- Kotlin Ktor 或 Spring Boot：与 Android/Kotlin 技术栈一致，长期可维护。
- PostgreSQL：账号、记录和同步元数据。
- Redis：验证码、限流、登录失败计数、Token 黑名单。
- S3 兼容对象存储：照片和备份媒体。
- Docker Compose：个人开发和早期部署。

如果想快速上线，也可以评估 Supabase 或 Firebase，但需要提前确认国内访问、成本、隐私和可迁移性。

### 认证

- Access Token + Refresh Token。
- Refresh Token 支持轮换。
- 服务端可撤销登录态。
- 邮箱验证码与密码登录共存。
- 账号体系保持邮箱验证码与密码登录，不接入第三方社交平台。

### 邮件服务

可选方向：

- 阿里云邮件推送、腾讯云 SES、SendGrid、Resend。
- 需要支持发送频控、模板、投递日志和退信处理。
