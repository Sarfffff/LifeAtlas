# 后续开发 TODO

本文档用于记录 V1.0 正式体验版之后的开发目标、优先级和风险点。后续体验反馈、UI 修改和功能改进都先进入本文档，再按模块拆分实现。

## 1. 当前基线

当前版本：`1.0.0`

当前状态：本地优先正式体验版。

已完成核心闭环：

```text
新增记录 -> 本地保存 -> 时间轴/地图展示 -> 详情查看与编辑 -> 备份导出与恢复
```

## 2. 近期优先级

### P0：体验反馈收集

- 安装 Debug APK 并完整体验 V1.0。
- 按页面记录 UI、交互、崩溃、卡顿和不符合预期的地方。
- 优先修复阻断使用的问题。
- 将非阻断问题归类为 UI 优化、交互优化、功能增强或后端能力。

### P0：登录与账号体系方案设计

登录能力不建议直接在 App 端硬写，需要先确定后端和账号安全策略。

目标能力：

- 邮箱验证码注册和登录。
- 邮箱验证码绑定与验证。
- 账号密码登录。
- 忘记密码与重置密码。
- 登录态保持和安全退出。
- 本地数据与账号数据的关联策略。
- 后续云同步的数据归属和冲突处理。

### P1：防恶意注册与登录安全

需要结合常见登录风险设计，不只做“能登录”。

建议策略：

- 邮箱验证码发送频率限制，例如同邮箱 60 秒内只能发送一次。
- 单邮箱每日验证码次数限制。
- 单 IP 每小时注册和验证码请求次数限制。
- 登录失败次数限制，连续失败后短时间锁定。
- 验证码有效期，例如 10 分钟。
- 验证码只存哈希，不明文保存。
- 密码使用强哈希算法，例如 Argon2id 或 bcrypt。
- 密码复杂度要求和弱密码拦截。
- 注册接口加入图形验证码或 Turnstile/hCaptcha 等人机验证。
- 设备指纹或匿名设备 ID 只用于风控，不用于过度追踪。
- 风险操作需要二次验证，例如修改邮箱、重置密码、删除账号。
- 服务端记录审计日志，便于排查异常注册和撞库攻击。

### P1：后端与云同步基础

如果接入登录，后端需要同时考虑数据同步。

建议拆分：

- 账号服务：注册、登录、邮箱验证、密码重置、Token 刷新。
- 用户资料服务：昵称、头像、偏好设置。
- 记录同步服务：记录、标签、照片元数据同步。
- 媒体服务：照片缓存或原图对象存储。
- 同步冲突策略：本地优先、服务端版本号、最后修改时间、软删除。
- 数据导出能力保持可用，避免用户被云端锁定。

### P1：UI 样式改版

后续 UI 修改建议集中成一轮设计迭代，不要零散修改。

建议方向：

- 统一首页、时间轴、地图、设置页的视觉语言。
- 重新设计记录卡片层级，突出照片、时间、地点和标签。
- 优化新增/编辑记录表单，让常用字段更顺手。
- 地图页增加更清晰的空状态和配置状态。
- 设置页按“数据、地图、账号、关于”分组。
- 适配深色模式。
- 增加统一 loading、错误、空状态组件。

### P1：账号接入后的 App 端页面

## 4. 技术选型建议

### 后端

推荐优先考虑：

- Kotlin Ktor 或 Spring Boot：与 Android/Kotlin 技术栈一致，长期可维护。
- PostgreSQL：账号、记录和同步元数据。
- Redis：验证码、限流、登录失败计数、Token 黑名单。
- S3 兼容对象存储：照片和备份媒体。
- Docker Compose：个人开发和早期部署。

如果想快速上线，也可以评估 Supabase 或 Firebase，但需要提前确认国内访问、成本、隐私和可迁移性。

### 认证

- Access Token + Refresh Token。
- Refresh Token 支持轮换。
- 服务端可撤销登录态。
- 邮箱验证码与密码登录共存。
- 后续可扩展第三方登录，但 V2.0 不建议一开始就做太多。

### 邮件服务

可选方向：

- 阿里云邮件推送、腾讯云 SES、SendGrid、Resend。
- 需要支持发送频控、模板、投递日志和退信处理。

## 5. 实现顺序建议

1. 体验 V1.0，记录 UI 和功能问题。
2. 完成 V1.1 本地体验增强，先不引入后端复杂度。
3. 编写账号与云同步技术设计文档。
4. 搭建后端最小服务：注册、登录、验证码、Token。
5. App 接入登录页和账号设置页。
6. 设计云同步数据模型。
7. 小范围测试同步和冲突处理。
8. 再考虑正式上线和高并发优化。
